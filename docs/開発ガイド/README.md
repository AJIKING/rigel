# 開発ガイド — rigel の進め方

rigel は **ハーネスエンジニアリング** と **TDD** の2本柱で開発する。
このフォルダはその「進め方（how）」をまとめたもの。「何を作るか（why/what）」は
[../mahjong-kifu-app-design.md](../mahjong-kifu-app-design.md)（設計ドキュメント＝単一真実源）を見ること。

エージェントの毎セッションの起点は、リポジトリ直下の [../../CLAUDE.md](../../CLAUDE.md)。

---

## このプロダクトの一言

実物の麻雀卓を撮った写真から **牌譜（盤面）のドラフトを自動生成**し、ユーザーが
確信度の低い箇所を修正・保存・共有できるサービス。**AI（Gemini）の読み取り精度が
サービスの成否を握る**ので、AI を取り囲む足場（ハーネス）の設計が中心になる。

## 1分サマリ

- **テストを書く前に実装を書かない。** Red → Green → Refactor。
- **スキーマが背骨。** 牌譜スキーマ（Zod）を全層が共有し、**AI出力は使う前に必ず検証する**。
- **AIに推測させない。** 読めない牌は `null` + `confidence` でスロットを保持。枚数・順序を壊さない。
- **`[決定]` と `[未確定]` を取り違えない。** 未確定を勝手に確定して実装を進めない。要実機検証は検証してから。
- **小さく進む。** 1タスク＝1つの振る舞い、縦切り（撮影→解析→ドラフト→修正→保存の「動く一筋」）。
- **完了の定義はゲート通過。** 型・lint・テスト全緑 + 信頼ゲート（推測しない / confidence / 課金整合 / 画像非保存）。
