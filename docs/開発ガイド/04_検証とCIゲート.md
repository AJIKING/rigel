# 04. 検証とCIゲート

ハーネスエンジニアリングの「検証ループ」を、具体的なコマンドとゲートに落とす章。
**完了＝ゲート通過**。「動いた気がする」では完了にしない。

---

## 検証コマンドを一本化する

エージェントが毎回迷わないよう、検証コマンドを**ここに一元化**する。
M0 でモノレポ（pnpm workspace / turborepo）を構築済み。スタック: TypeScript 一気通貫 / Vitest / ESLint / Prettier。

### モノレポ全体（リポジトリ直下）

ルートの `package.json` スクリプトは turbo 経由で全パッケージに展開される。

```bash
pnpm install                 # 依存
pnpm typecheck               # turbo → 各パッケージ tsc --noEmit（型は最重要ゲート）
pnpm lint                    # turbo → ESLint（警告ゼロ）
pnpm format:check            # Prettier 差分ゼロ（Markdown は .prettierignore で除外＝手整形）
pnpm test                    # turbo → Vitest（unit + integration）
pnpm build                   # turbo → ビルド（schema の dist など）
```

### アプリ別（参考。`web`/`mobile` は M5+ で追加）

```bash
# packages/schema … スキーマ＝背骨。ここのテストは特に厚く
pnpm --filter @rigel/schema test

# apps/api (Cloudflare Workers) … Miniflare/wrangler で D1 を含めて検証（M5/M7）
pnpm --filter api test
pnpm --filter api exec wrangler types     # D1/bindings の型（バインディング追加後）

# apps/web (Next.js) / apps/mobile (Expo) … 追加後
# pnpm --filter web build / pnpm --filter mobile exec tsc --noEmit
```

> 上記が**ローカルゲート**。コミット前にこれを全部緑にする。CI でも同じものを回す（ローカルとCIで同一コマンド）。

> **CI**: `.github/workflows/ci.yml` で同じゲート（install → typecheck → lint → format:check → test → build）を
> Ubuntu / Node 22 / pnpm 10 で実行する。`pnpm install --frozen-lockfile` を使うので `pnpm-lock.yaml` をコミットすること。

## ゲート（CI で必須通過させるもの）

| ゲート | 内容 | 失敗時 |
|---|---|---|
| **typecheck** | `tsc --noEmit` でエラーゼロ（スキーマの型が全層に効く） | マージ不可 |
| **lint** | ESLint 警告ゼロ | マージ不可 |
| **format** | Prettier 差分ゼロ | マージ不可 |
| **test** | 全テスト緑（Vitest） | マージ不可 |
| **trust（信頼ゲート）** | AI出力を扱う変更は「Zod検証・推測しない・confidence・課金整合・画像非保存」のテストが存在（下記） | マージ不可 |
| **build** | 各アプリ/パッケージがビルドできる | マージ不可 |

ハーネス原則「観測可能なガバナンス」に従い、**正しさの規則はゲートで強制**する（注意書きではなく）。

---

## 信頼ゲート（rigel 固有・最重要）

zoma の「安全装置ゲート（バックアップ/差分/取り消し）」に相当するのが、rigel ではこの**信頼ゲート**。
AI の出力や課金・プライバシーに触れるコードは、以下を満たさないとマージしない。

- [ ] **AI 出力は使う前に必ず Zod 検証**（`AiRiverResponseSchema` / `AiHandResponseSchema` / 最後に `KifuSchema`）— 検証を通すテストがある
- [ ] **推測で埋めない** — 読めない牌は `tile: null` + `confidence: 0` でスロットを残し、枚数・`order` 連番が壊れないテストがある
- [ ] **confidence を全牌に持たせ**、低 confidence が UI で要確認ハイライトになる（人手修正の起点）テストがある
- [ ] **混在パーツの JSON 抽出**（Code Execution 使用時）が、種類でパーツを仕分けて最終 JSON を取り出せるテストがある
- [ ] **課金カウントは成功時のみ +1**、競合で二重加算・取りこぼしが無いテストがある
- [ ] **撮影画像を永続化しない**（保存されるのは解析後の `Kifu` JSON のみ）
- [ ] `[未確定]` を**確定扱いで実装していない**（実機検証で確定し、設計ドキュメントを `[決定]` に更新済み）

> このゲートが通らない限り、その機能は「未完成」。便利機能より優先する。

## eval（AI 精度の評価ループ／CIの赤緑とは分離）

AI の読み取り精度は非決定的なので、**合否ゲートには載せず**、評価用データセットで継続評価する
（ハーネス: Evals & Verification）。詳細は [02_TDD開発ガイド.md](02_TDD開発ガイド.md)。

最小構成の考え方：

- **ラベル付きテスト画像 20–30枚**（河4分割＋正立後の各方向／手牌）と**正解牌譜**をフィクスチャ化。
- 測る3指標：①牌単位の正解率 ②**「confidence 高いのに誤読」率（最重要）** ③順序・リーチ牌の正解率。
- 「素 Flash」と「Code Execution 有り」の A/B でコストと精度を比べ、`[未確定]`（Agentic Vision の要否）を潰す。
- 退行検知：過去に誤読した牌・条件を eval ケースに固定する（regression として残す）。

> 指標の優先順位：**自信満々の誤読を出さない > 正しく読む > 人が直しやすい > 速い・安い**。この順を逆転させない。

## 観測・トレース

- **AI Gateway**（Cloudflare）を AI 呼び出しの手前に噛ませ、キャッシュ・コスト監視・流量制御・レート制限対策を効かせる。
- confidence の分布・null 率・3指標をログ/集計し、後から「なぜこの読み取りか」を再構成できるようにする。
- テスト失敗時は、落ちたケース名と期待／実際の値が一目で分かる出力にする（[02](02_TDD開発ガイド.md) 参照）。

## 報告フォーマット（タスク完了時）

エージェントは完了報告に以下を含める。

```
## 何を
（実装した振る舞いを1〜2行）

## どう確認したか
- pnpm -r typecheck   → エラーなし
- pnpm -r test        → PASS（n件）
- 信頼ゲート          → Zod検証 / 推測しない / 課金整合 などの該当テスト緑
（AI精度に関わるなら）eval 3指標の数値 と A/B 結果

## 影響範囲
（触ったパッケージ/アプリ・新規ファイル・依存追加の有無・設計ドキュメントの[決定]更新有無）

## 残課題 / 次の一手
（あれば。残った [未確定] は明示する）
```

憶測で「できました」と言わず、**実際のコマンド出力**を根拠にする。
