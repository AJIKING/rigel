# 03. タスク分解とPlan運用

ハーネスエンジニアリングの「計画アーティファクト」を、rigel の開発に落とす章。
**非自明なタスクは、着手前に Plan を出して合意する。**

---

## なぜ計画をファイルにするか

- エージェントの作業は**計画の質に強く依存する**（ハーネス原則：計画アーティファクト）。
- チャット内の合意は消える。Plan / Task をファイルに残すと、セッションをまたいで再利用でき、レビューできる。
- 「何を作らないか」を先に決められる（スコープ漏れ・暴走の防止）。設計ドキュメント 2章の**非スコープ**を Plan に引き写す。

テンプレ：[../templates/Plan.md](../templates/Plan.md) ／ [../templates/Task.md](../templates/Task.md)

---

## 2段階：Plan → Task

```
Plan（設計図） ──分解──▶ Task（実行単位） ──TDD──▶ 実装
   1機能/1マイルストーン      1つの振る舞い        Red→Green→Refactor
```

- **Plan**：1つの機能やマイルストーン。全体像・影響範囲・どのハーネス構成要素か・検証/eval方針・受け入れ条件を書く。
- **Task**：Plan を「1つの振る舞い＝1 TDDサイクル」に割ったもの。これが実装の最小単位。

---

## 良いタスクの条件（INVEST 風）

| 観点 | 中身 |
|---|---|
| 独立 | 他タスクの完了を待たずに着手・検証できる |
| 小さい | 1つの振る舞い。1〜数サイクルの TDD で終わる |
| テスト可能 | 「どうなれば完了か」をテストで書ける（AI精度は eval で別途） |
| 縦切り | 層をまたいでも「動く一筋」で切る（スキーマだけ・UIだけ、に水平分割しない） |
| 確実 | `[未確定]` を含むなら、その検証（実機確認・A/B）をタスクに含めてから本実装する |

> **悪い例**：「解析パイプラインを実装する」（大きすぎ・曖昧）
> **良い例**：「`AiRiverResponse` を `toAbsoluteSeat` で絶対席に写し、`Kifu.seats` に河を詰める」（1つの振る舞い・テスト可能）

---

## `[決定]` / `[未確定]` を分解の基準にする（rigel 固有）

設計ドキュメントは各項目に `[決定]` / `[未確定]` を付けている。**ここを分解の出発点にする。**

- `[決定]` 項目 → そのまま受け入れ条件・テストに落とす。
- `[未確定]` 項目 → **本実装の前に「検証タスク」を1つ置く**（実機確認 / A/B / 精度実測）。検証で確定するまで、依存する本実装の期待値を確定しない。
- 検証で出た結論は、設計ドキュメントの当該箇所を `[決定]` に更新してから先へ進む（記憶＝足場に残す）。

> 例：`toAbsoluteSeat` の回転方向（`[未確定／要実機検証]`）は、「東家を手前に置いた写真で right/top/left が南/西/北か目視確認」を**先に1タスク**として済ませ、結論をテスト期待値に焼き付ける。

---

## 分解の手順

1. **設計ドキュメントとCLAUDE.mdで文脈確認** — 触る領域がどのハーネス構成要素か（[01](01_ハーネスエンジニアリング.md) の対応表）を特定。`[決定]/[未確定]` を仕分ける。
2. **Plan を書く** — テンプレに沿って全体像・非対象・影響範囲・検証/eval方針・受け入れ条件。
3. **振る舞いに割る** — スキーマ検証・変換・正常系・異常系（null/混在パーツ）・境界値を列挙し、各々を Task にする。
4. **順序づけ** — `[未確定]` の検証 → スキーマ/変換 → 解析本体 → UI → 保存 → 課金、の順を基本にする。背骨と検証を先に。
5. **着手前に合意** — Plan を提示し、人間の承認を得てから Red を書き始める。

---

## マイルストーン（叩き台）

設計ドキュメント 10章「推奨開発順序」より。**AI精度がサービスの成否を握る**ので、認証・課金は最後に外側として被せる。
実装着手時に各 M を Plan として具体化する。

1. **M0 土台** — モノレポ（pnpm workspace / turborepo）、TypeScript、lint/format/test と CI ゲートの整備。
2. **M1 スキーマ確定** — 牌譜スキーマを `packages/schema/src/index.ts` として確定（済）。型エクスポート・`toAbsoluteSeat`・基本テスト。
3. **M2 相対→絶対変換の検証**（`[未確定]`潰し）— `CAMERA_ORDER` の回転方向を実機確認し、`toAbsoluteSeat` の期待値をテストに焼き付ける（TODO#1）。
4. **M3 河プロンプトの単方向化** — `river_reader_prompt.md` を1方向版に直し、`AiRiverResponse` 形式に合わせる。
5. **M4 AI精度の実地テスト / eval** — ラベル付きテスト画像20–30枚で3指標を測定。河4分割＋正立後、素Flash と Code Execution を A/B（TODO#2・#3）。← **最優先で固める**。
6. **M5 解析パイプライン（Workers）** — 4分割＋正立 → 8画像 → Gemini → Zod検証 → 相対絶対変換 → Kifu組み立て → `KifuSchema` 最終検証。混在パーツの JSON 抽出を含む。
7. **M6 牌譜描画UI** — `packages/ui` に SVG 描画・confidence ハイライト・修正操作。
8. **M7 保存/閲覧（D1）** — `users` / `game_logs`、画像非保存、保存済み牌譜の閲覧。
9. **M8 認証＋回数カウント＋課金** — Google認証、解析成功時のみ加算、カウント整合性、月額。

> この順序は**スキーマ（背骨）と AI 精度を最初に**置いているのが肝（[01](01_ハーネスエンジニアリング.md) の「まず足場を直す」）。
> 認証から作り始めない。

## 計画と TDD の接続

Plan の「受け入れ条件」は、そのまま**最初に書く失敗テスト（Red）の集合**になる。

```
受け入れ条件:
- 不正な牌コードを含む AI 出力は AiRiverResponseSchema.parse が拒否する
        ↓ そのまま
Red: it("不正な牌コードを弾く", ...) を書いて失敗を確認
        ↓
Green: スキーマ検証を噛ませる最小実装で通す
```

`[未確定]` 由来の受け入れ条件は、**実機検証で期待値を確定してから** Red を書く。

→ TDD の詳細は [02_TDD開発ガイド.md](02_TDD開発ガイド.md)、ゲートは [04_検証とCIゲート.md](04_検証とCIゲート.md)。
