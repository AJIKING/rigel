# 01. ハーネスエンジニアリング — rigel への適用

> 参考: [awesome-harness-engineering](https://github.com/ai-boost/awesome-harness-engineering)

## ハーネスエンジニアリングとは

**ハーネス（harness）= AIを取り囲む足場**のこと。
具体的には次のような構成要素を指す。

- **文脈の供給**（何をモデルに見せるか）
- **ツールのインターフェース**（何をどう操作させるか / 入出力スキーマ）
- **計画アーティファクト**（Plan / Task など、作業の設計図）
- **検証ループ**（テスト・eval・ゲート）
- **記憶**（セッションをまたぐ永続知識）
- **サンドボックス／権限**（どこまで触ってよいか）

ハーネスエンジニアリングは「モデルそのもの」ではなく、この**足場を設計する規律**。
中核の主張は **「モデルではなく、ハーネスが成否を分ける」**。
各構成要素は「モデル単体ではその役割を果たせないから」存在している。

### 5つの原則

1. **モデル非依存設計** — 足場はモデルが賢くなれば不要になる一時的な scaffolding。永続アーキテクチャと混同しない。
2. **インターフェース＝UX** — ツール設計・権限・文脈供給は「モデルにとっての UX 問題」。出力スキーマも UX。
3. **合成可能なプリミティブ** — 4分割／正立化／単方向プロンプト／検証／変換などの**部品の組み合わせ**。巨大な一枚岩プロンプトにしない。
4. **経験的最適化** — 「モデルを変える」より「文脈の整理・前処理・出力スキーマ設計」の方が効くことが多い。まず足場を直す。
5. **観測可能なガバナンス** — 正しさの規則（推測しない・検証する）は**プロンプトの注意書きで済ませず、コードとテストで強制・監査可能**にする。

## なぜ rigel に効くのか

rigel の価値は「写真 → 牌譜ドラフト」の**読み取り精度**にある。だが Gemini に河の写真を
そのまま投げても精度は出ない（実画像テストで top/left/right 方向は回転・遠近・密集で精度が落ちる）。
**ここがハーネスの出番**だ。rigel のコードのほとんどは「モデルの周りの足場」である。

> **モデル（Gemini）は1つの部品にすぎない。** 4分割＋正立で全方向を好条件に揃え、
> 単方向プロンプトでタスクを単純化し、Zod で出力を検証し、confidence で人手修正に橋渡しし、
> 精度 eval で退行を検知する。**これら足場が rigel の精度＝成否を決める。**

だから原則4「まず足場を直す」が文字どおり効く。読み取りが甘いとき、最初に変えるのはモデルではなく
**前処理（分割・正立・解像度）・プロンプト（単方向・記法固定・手順分割）・出力スキーマ（confidence・null許容）** である。

## 構成要素を rigel にマッピング

| ハーネス構成要素 | rigel での実体 |
|---|---|
| 文脈供給 / 前処理 | 河1枚を**4分割＋正立化**して bottom と同条件に揃える。解像度調整。手牌は撮影時点で正立 |
| ツール設計（入出力スキーマ） | **牌譜スキーマ（Zod）** が全層共有の背骨。`AiRiverResponse` / `AiHandResponse` が AI 出力の型 |
| 計画アーティファクト | [Plan.md](../templates/Plan.md) / [Task.md](../templates/Task.md)。設計ドキュメントの `[未確定]` を潰す検証計画 |
| 検証ループ | TDD（Red-Green-Refactor）+ CIゲート + **AI精度 eval**（ラベル付きテスト画像） |
| 記憶 / 状態 | 設計ドキュメント・決定ログ・CLAUDE.md。保存する牌譜は解析後の `Kifu` JSON のみ |
| 権限 / human-in-the-loop | **confidence の低い牌をハイライトして人に修正させる**。推測で確定させない |
| 観測 / トレース | AI Gateway（キャッシュ・コスト監視・流量制御）、confidence 分布、精度3指標 |
| サンドボックス / 制約 | **撮影画像を保存しない**（解析後 JSON のみ）。月◯件の回数制限がコスト・流量の境界 |

この対応表は、**実装の各モジュールがどのハーネス原則に対応するか**の地図でもある。
新機能を作るときは「これはどの構成要素か」を最初に決める（Plan のテンプレに項目がある）。

## rigel 固有のリスクと足場（zoma との違い）

参考にした zoma はユーザーの実ファイルを破壊的に書き換えるため、足場の主役は
**バックアップ／差分／取り消し**だった。rigel は**ユーザーのファイルを破壊的に変更しない**
（撮影画像すら保存しない）。だから rigel の「壊さない」は意味が違う。守るべき不変条件は次の4つ。

1. **誤読を断定しない。** 読めない牌は推測で埋めず `tile: null` + `confidence: 0` でスロットを残す。
   最重要指標は「**confidence が高いのに誤読**」率（自信満々の誤りは人が見逃す）。
2. **スキーマを単一真実源にする。** 牌譜スキーマ（Zod）を全層（RN / Next.js / Workers / AI出力検証）が共有し、
   **AI 出力は使う前に必ず Zod で検証**する。型のズレを境界で止める。
3. **`[決定]` と `[未確定]` を取り違えない。** 設計ドキュメントの `[未確定]`（例: `toAbsoluteSeat` の回転方向、
   Agentic Vision の要否）を**勝手に確定して実装しない**。要実機検証はテスト／検証で潰してから進む。
4. **課金とプライバシーを誤らない。** 解析カウントは**成功時のみ +1**（整合性に注意）。**撮影画像は永続化しない**。

> これらは「壊さないでね」という注意書きではなく、**Zod 検証・テスト・eval・カウント整合テストで強制**する
> （原則5: 観測可能なガバナンス）。

## 実務ルール（このリポジトリでの具体）

- **足場は口頭ではなくファイルに残す。** 決定・規約・前提は CLAUDE.md か docs に書く。チャット内だけの合意は消える。
- **文脈は絞って渡す。** タスクに関係するファイル（設計ドキュメントの該当章・スキーマ・対象アプリ）だけをスコープに入れる。
- **出力スキーマで縛る。** AI には「指定 JSON のみ」を返させ、confidence と null を必須にする。プロンプトの語りで品質を担保しない。
- **正しさは実行可能にする。** 「推測しないでね」ではなく、null スロット保持・Zod 検証・精度 eval を**コードとテストで**強制する。
- **まず足場を疑う。** 読み取りがうまくいかないとき、モデルを責める前に「前処理（分割・正立・解像度）・プロンプト・出力スキーマ」を見る（原則4）。
- **モデル名をハードコードしない。** Gemini のモデル名は AI Studio で現行の対応モデルを確認して使う（足場は捨てられる前提）。

## アンチパターン

- ❌ 河を1枚のまま巨大プロンプトで全方向まとめて読ませる（→ 4分割＋正立＋単方向プロンプトに分ける）
- ❌ AI に席（東南西北）を推測させる（→ カメラ相対 bottom/right/top/left で出させ、アプリ側で絶対変換）
- ❌ 読めない牌を「それっぽい牌」で埋める（→ null + confidence でスロット保持。推測は退行の温床）
- ❌ AI 出力を検証せず使う（→ 使う前に必ず Zod 検証。スキーマが単一真実源）
- ❌ `[未確定]` を勝手に確定して実装を進める（→ 検証で潰してから。誤ると全席90°ズレ等の事故）
- ❌ 「モデルが悪い」で止まる（→ 前処理・プロンプト・出力スキーマの不足をまず疑う）
