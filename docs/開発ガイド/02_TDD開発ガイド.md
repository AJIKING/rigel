# 02. TDD開発ガイド

rigel は **テスト駆動開発（TDD）** で進める。
全層が TypeScript で、中心には**牌譜スキーマ（Zod）という1つの背骨**がある。
だから「スキーマに対して正しいか」「AI の不確実な出力を安全に捌けるか」をテストで保証することが品質の中心。

---

## 鉄則

> **失敗するテストを書く前に、本番コードを書かない。**

理由：

- rigel の価値は「写真 → 信頼できる牌譜ドラフト」。AI は誤読する前提なので、**「誤読・null・席変換・課金境界がどう振る舞うべきか」をテストで先に定義しないと正しく作れない**。
- 開発エージェント（Claude Code / Codex）にとって、テストは**正しさの仕様＝期待する振る舞いの明示**。曖昧な指示よりテストの方が伝わる（ハーネス原則：検証ループ）。

---

## サイクル：Red → Green → Refactor

```
┌─ Red ──────────────┐   失敗するテストを1つ書く。
│ 期待する振る舞いを   │   実行して「ちゃんと失敗する」ことを確認する。
│ テストで宣言する     │   （失敗しないテストはバグ）
└─────────┬──────────┘
          ▼
┌─ Green ────────────┐   テストを通す「最小の」実装を書く。
│ とにかく緑にする     │   きれいさより通すこと優先。重複・仮実装OK。
└─────────┬──────────┘
          ▼
┌─ Refactor ─────────┐   テストを緑に保ったまま整える。
│ 重複除去・命名・構造 │   振る舞いを変えない。緑が崩れたら戻す。
└────────────────────┘
```

1サイクル＝**1つの振る舞い**。大きいタスクは [03_タスク分解とPlan運用.md](03_タスク分解とPlan運用.md) で振る舞い単位に割る。

---

## スキーマ起点で進める（rigel の肝）

牌譜スキーマ（[../../packages/schema/src/index.ts](../../packages/schema/src/index.ts)）が**全ての中心＝背骨**。
推奨開発順序（[03](03_タスク分解とPlan運用.md)）も「スキーマ確定 → 縦に1本通す」。テストもここから始める。

- **AI 出力は使う前に必ず `*.parse()` で検証**する。検証を通っていない生レスポンスを下流に流すコードは書かない。
- 変換関数（`toAbsoluteSeat` など）は純粋関数なので**最も書きやすく・最も事故りやすい**。table-driven で網羅する。
- `null`（読めない牌）・`confidence`・`riichi`・スロット保持（枚数/順序）を**境界値として必ずテスト**する。

---

## テストの層（テストピラミッド）

| 層 | 何を検証 | 主なツール |
|---|---|---|
| **Unit** | スキーマ検証 / `toAbsoluteSeat` / JSONパーツ仕分け / confidence・null 処理 | Vitest |
| **Integration** | 解析パイプライン（fake Gemini → Zod検証 → 相対絶対 → Kifu組み立て）、D1リポジトリ、カウント整合 | Vitest（+ Workers は Miniflare/wrangler） |
| **Component / UI** | 牌のSVG描画・confidenceハイライト・修正操作 | Vitest + Testing Library（RN / Web） |
| **Eval（非決定的・別枠）** | AI の読み取り精度（ラベル付きテスト画像） | 専用 eval ランナー（CIの合否ゲートとは分離） |
| **E2E（最小限）** | 撮影 → 解析 → ドラフト → 修正 → 保存 の主要フロー | 後段で薄く |

**比率の目安**：Unit を厚く、Integration を要所に、E2E は主要フローだけ薄く。
**AI 精度は「テスト」ではなく「eval」**として分ける（下記）。非決定的なものを CI の赤/緑に混ぜない。

---

## rigel 特有のテスト方針

### AI 出力は「壊れた入力」を必ずテストする
Gemini は仕様どおりに返さないことがある。本番コードの前に、次のような異常系のテストを書く。

1. **スキーマ違反を弾く** — 不正な牌コード・欠けたフィールド・型違いを `AiRiverResponseSchema.parse` が拒否する。
2. **null スロットを保持する** — `tile: null, confidence: 0` を含む出力でも、枚数と `order` の連番が壊れない。
3. **混在パーツから JSON を拾う**（Agentic Vision / Code Execution 使用時）— レスポンスが「テキスト＋生成コード＋実行結果」の混在で返るケースを、**位置でなく種類でパーツを仕分け**、最終テキストから JSON を抽出して `parse` できる。（設計ドキュメント 4章 `[未確定]` 参照）

### 相対→絶対変換は遷移表でテストする（最重要級）
`toAbsoluteSeat(camera, bottomSeat)` は**誤ると全席が90°ズレる**。`bottom × {bottom/right/top/left}` の
全組み合わせを table-driven で固定する。設計ドキュメントの `[未確定／要実機検証]`（`CAMERA_ORDER` の回転方向）は
**実機で1回確認してから**期待値を確定し、その期待値をテストに焼き付けて退行を防ぐ。

### confidence は「人手修正への橋」としてテストする
- 低 confidence の牌が UI で**要確認ハイライト**になること（human-in-the-loop の起点）。
- ハイライトされた牌を人が直すと `Kifu` が更新され、最終的に `KifuSchema.parse` を通ること。

### 課金カウントは整合性をテストする
- 解析カウントは**成功時のみ +1**、失敗時は消費しない。
- 「確認 → 解析 → 加算」の競合（同時実行）で二重加算・取りこぼしが起きない（D1トランザクション / Durable Objects）。

### 外部依存（Gemini API / AI Gateway）は境界でモックする
Gemini 呼び出しはインターフェース化し、テストでは**記録済みレスポンス or フェイク**を差す。
正常系・スキーマ違反・混在パーツ・タイムアウトをフェイクで再現する。**実 API を叩くテストは eval（別枠）に限定**する。

## テストの書き方（Vitest の例：table-driven）

```ts
import { describe, it, expect } from "vitest";
import { toAbsoluteSeat } from "@rigel/schema";

describe("toAbsoluteSeat（カメラ相対→絶対席）", () => {
  // ⚠️ 期待値は実機で CAMERA_ORDER の向きを確認してから確定する（設計ドキュメント 5章）
  const cases = [
    { name: "東家が手前なら右はそのまま南", camera: "right", bottom: "east",  want: "south" },
    { name: "東家が手前なら向かいは西",     camera: "top",   bottom: "east",  want: "west"  },
    { name: "東家が手前なら左は北",         camera: "left",  bottom: "east",  want: "north" },
    { name: "南家が手前なら向かいは北",     camera: "top",   bottom: "south", want: "north" },
  ] as const;

  it.each(cases)("$name", ({ camera, bottom, want }) => {
    expect(toAbsoluteSeat(camera, bottom)).toBe(want);
  });
});
```

ポイント：

- テスト名は**日本語で振る舞いを述べる**（仕様書として読める）。
- 1ケース1アサート方針に近づけ、落ちたとき原因が一目で分かるようにする。
- まず失敗（Red）を確認 → 実装（Green）→ 整える（Refactor）。

## AI 精度の eval（テストとは別枠）

AI の読み取り精度は非決定的で、CI の赤/緑ゲートには載せない。**評価用データセット**で別途回す。

- **ラベル付きテスト画像 20–30枚**（正解牌譜つき）をフィクスチャ化（設計ドキュメント 4章）。
- 測る3指標：①牌単位の正解率 ②**「confidence 高いのに誤読」率（最重要）** ③順序・リーチ牌の正解率。
- A/B 比較に使う：河4分割＋正立後、**素の Flash と Code Execution 有り**で精度・コストを比べ `[未確定]` を潰す。
- 退行検知：過去に誤読した牌・条件を eval ケースに固定する（regression として残す）。

> 指標の優先順位：**自信満々の誤読を出さない > 正しく読む > 人が直しやすい > 速い・安い**。この順を逆転させない。

## やってよいこと / だめなこと

- ✅ 仮実装・ベタ書きで一旦 Green にする（Refactor で消す）
- ✅ テスト名で意図を語る／境界値（null・confidence 0・空配列・連番の穴）・異常系を足す
- ✅ AI 出力は正常系だけでなく**スキーマ違反・混在パーツ・タイムアウト**もフェイクでテスト
- ❌ テスト無しで本番コードを足す
- ❌ 失敗を確認せずに Green と言う
- ❌ AI 出力を `parse` せずに下流へ流す（スキーマ検証を飛ばさない）
- ❌ `[未確定]` を勝手に確定して期待値を書く（実機検証で確定してから焼き付ける）
- ❌ 非決定的な AI 精度を Unit テストにして CI を不安定にする（eval に分ける）

---

## 「完了」の定義（Definition of Done）

1タスクは、次を**すべて**満たして初めて完了。

- [ ] 振る舞いごとに Red → Green → Refactor を回した
- [ ] 関連テストが全部緑（出力を報告に添える）
- [ ] 型チェック（`tsc --noEmit`）・lint・format を通した（[04_検証とCIゲート.md](04_検証とCIゲート.md)）
- [ ] AI 出力を扱うなら「スキーマ検証・null スロット保持・推測しない」のテストがある
- [ ] 課金/プライバシーに触れるなら「成功時のみ加算」「画像を保存しない」のテストがある
- [ ] 影響範囲と確認方法を報告に書いた
