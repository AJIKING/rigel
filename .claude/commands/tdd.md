# TDD サイクル

rigel の鉄則: **失敗するテストを書く前に本番コードを書かない。** 1サイクル＝1つの振る舞い。

実装する振る舞い: $ARGUMENTS

参照: @docs/開発ガイド/02_TDD開発ガイド.md / @CLAUDE.md

## 進め方（この順を厳守）

1. **文脈確認** — 対象パッケージ/アプリと、これがどのハーネス構成要素かを把握する。スキーマ（背骨）に触れるか確認。
2. **Red** — 振る舞いを宣言する失敗テストを1つ書く。
   - Vitest、table-driven（`it.each`）、テスト名は日本語で振る舞いを述べる。
   - AI 出力を扱うなら、正常系だけでなく **null スロット・混在パーツ・スキーマ違反** も境界として書く。
   - 実行して **「ちゃんと失敗する」ことを確認**（失敗しないテストはバグ）。
3. **Green** — 通す最小実装だけを書く（仮実装・ベタ書き可）。テストが緑になるまで。
4. **Refactor** — 緑を保ったまま重複除去・命名・構造を整える。振る舞いは変えない。
5. **信頼まわり**（AI出力/課金/プライバシーに触れるなら必須） — Zod検証(`*.parse`)・推測しない(null+confidence のスロット保持)・成功時のみ加算・画像非保存 を、テストとして追加する。
6. **ゲート** — `/gate` 相当（typecheck・lint・format・全テスト緑）を通す。AI精度は eval で別途。
7. **報告** — 何を/どう確認したか（テスト出力つき）を述べる。憶測で「できた」と言わない。

> `[未確定]`（例: `toAbsoluteSeat` の回転方向、Agentic Vision 要否）を**勝手に確定して期待値を書かない**。
> 実機検証で確定してからテストに焼き付ける。

各ステップを飛ばさず、Red の失敗確認と Green の緑確認を**実際のコマンド出力で**示すこと。
