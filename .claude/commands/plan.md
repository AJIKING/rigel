# Plan 作成

rigel の開発ガイドに従い、**着手前に Plan を出して合意する**。実装はまだ書かない。

対象: $ARGUMENTS

## 手順

1. 文脈を確認する:
   - @CLAUDE.md（ハードルール・作業ループ）
   - @docs/mahjong-kifu-app-design.md（why/what の単一真実源。`[決定]`/`[未確定]` を仕分ける）
   - @docs/開発ガイド/03_タスク分解とPlan運用.md（分解の作法）
   - @packages/schema/src/index.ts（背骨。触る型・変換を把握）と関連する既存コード（あれば）
2. @docs/templates/Plan.md の形式で Plan を作成する。特に次を必ず埋める:
   - **ハーネス構成要素**（文脈/前処理・ツール(スキーマ)・計画・検証/eval・記憶・権限HITL・観測・制約のどれか）
   - **スコープ（やらないこと含む）** — 設計ドキュメント 2章の非スコープを引き写す
   - **`[決定]`/`[未確定]` の仕分け** — 触れる `[未確定]` は本実装の前に検証タスクを置く
   - **信頼まわり**（AI出力/課金/プライバシーに触れるなら: Zod検証・推測しない(null+confidence)・成功時のみ加算・画像非保存。無いなら「該当なし」明記）
   - **受け入れ条件** = そのまま最初の失敗テスト(Red)にできる粒度で
   - **Task 分解**（1つ＝1つの振る舞い。`[未確定]`検証 → スキーマ/変換 → 本体 → UI → 保存 → 課金 の順）
3. Plan を提示し、**人間の承認を求める**。承認前に実装・テストを書き始めない。
4. 承認されたら、必要に応じて `docs/plans/` に Plan を保存し、最初の Task は `/tdd` で着手する。

非自明でないタスクなら、その旨を述べて簡潔な Plan に留めてよい。
